01 — The Challenge

Flash Attention on a GPU
without datacenter features

The RTX 5080 is NVIDIA's consumer Blackwell chip — fast tensor cores, but none of the datacenter magic that makes H100 and B200 kernels fly. No WGMMA (warp group MMA), no TMA (tensor memory accelerator), no warp specialization barriers. Just raw CUDA, shared memory, and 84 streaming multiprocessors.

Flash Attention is memory-bound by nature: the softmax in the inner loop breaks the pure GEMM pattern that tensor cores crave. The question was — how close to peak can we get with only the tools available on consumer silicon?

⚡

234.8 TFLOPS Peak

The RTX 5080's theoretical FP16 throughput. Our target ceiling. Production Flash Attention on H100 achieves 50-70% of peak — could we match that ratio?

🧱

No WGMMA / TMA

sm_120 (consumer Blackwell) is architecturally closer to Ada Lovelace than datacenter Blackwell. 32-thread warps, not 128-thread warp groups. Manual memory management.

🔥

The Softmax Wall

Every KV tile needs a full softmax: find max, compute exp, sum, normalize. This serial dependency between tiles is what makes attention fundamentally harder than GEMM.

02 — The Optimization Journey

Five versions, five breakthroughs

Each version identified and eliminated a specific bottleneck. The profiler told us where time was being wasted; the architecture told us how to fix it.

V1 — BASELINE
Scalar FP32 Attention
No tensor cores. Each thread computes one element of Q×K^T using scalar multiply-add. The entire 64×64 attention matrix computed element-by-element. The GPU's tensor cores sit completely idle.
~2.7 TFLOPS
V3 — TENSOR CORES
WMMA Fragment Operations
Switched to nvcuda::wmma — NVIDIA's C++ API for tensor cores. Each warp now computes 16×16 matrix tiles in a single instruction. 10× faster compute, but we immediately hit a new wall: global memory bandwidth.
26.7 TFLOPS · 10× from baseline
V6 — VECTORIZED LOADS
uint4 Coalesced Memory Access
Replaced individual half-precision loads with 128-bit uint4 vector loads. Each thread now moves 16 bytes per transaction instead of 2. Global memory bandwidth utilization jumped from ~15% to ~40%.
38.3 TFLOPS · +43% from WMMA
V7 — RAW PTX
PTX MMA + ldmatrix Instructions
Replaced WMMA's opaque fragment API with raw PTX assembly. mma.sync.aligned.m16n8k16 gives us known register layouts — each thread holds exactly 4 floats at deterministic (row, col) positions. ldmatrix.sync loads tiles directly from shared memory into MMA-ready registers.
49.2 TFLOPS · Known register layout unlocked
V8 — IN-REGISTER SOFTMAX
The Breakthrough: Eliminating smem_s
The key innovation. S = Q×K^T results stay in registers — never written to shared memory. Softmax computed via intra-warp shuffles + tiny cross-warp exchange (1KB vs 16KB). Eliminated the largest shared memory round-trip and one synchronization barrier per tile.
125.2 TFLOPS · 53% of peak · 47× from baseline

04 — The Key Insight

Why in-register softmax
changes everything

In a standard flash attention kernel, the S = Q×K^T matrix is written to shared memory (16KB), a barrier fires, then softmax reads it back. This round-trip dominates the inner loop.

Before: smem_s Round-Trip

  • MMA writes S to shared memory (16KB)
  • __syncthreads() — all 8 warps stall
  • Each warp reads 8 rows from smem_s
  • Compute softmax per row
  • Write P to smem_p (9KB)
  • __syncthreads() again
  • 4 barriers per KV tile
  • ~54KB shared memory

After: Register Softmax

  • MMA results stay in s_acc[4][4] registers
  • Shuffle reduce across 4 threads per row
  • Exchange 128 floats between warp halves
  • exp() computed in registers
  • Write P to smem_p directly
  • Only 3 barriers per KV tile
  • ~38KB shared memory
  • Tensor cores active 50% (was 18%)

05 — The Bug That Nearly Stopped Us

ldmatrix.x2.trans:
when half the matrix goes missing

PTX's ldmatrix.sync.aligned.m8n8.x2.trans loads two 8×8 matrices. Threads 0-7 provide addresses for matrix 0, threads 8-15 for matrix 1. Our code used lane_id % 8 for both groups — making threads 8-15 load identical data to threads 0-7.

The result: only half the K-dimension was loaded, duplicated into both halves. Rows 0-7 of the output looked correct (they only needed the first half), but rows 8-15 were corrupted. The identity matrix test made it obvious — row 8 showed values 1-8 instead of 65-72.

// BUG: threads 8-15 duplicate threads 0-7 int k_row = lane_id % 8; ldmatrix_x2_trans(b0, b1, smem_k + (ni * 8 + k_row) * KV_STRIDE + ki * 16); // FIX: matrix 1 reads the next 8 columns int k_row = lane_id % 8; int mat = (lane_id / 8) % 2; ldmatrix_x2_trans(b0, b1, smem_k + (ni * 8 + k_row) * KV_STRIDE + ki * 16 + mat * 8);

At 8.5% tensor core utilization, the MMA units were idle 91% of the time — starved by shared memory traffic and synchronization barriers. 07 — What Consumer Blackwell Can't Do

The datacenter features
we worked without

Examining DeepSeek's production kernels for H100 (sm_90) and B200 (sm_100) revealed an entirely different class of hardware. Here's what was available to them but not to us.

🔄

WGMMA

Warp Group MMA — 128 threads issue a single matrix multiply. B operand reads directly from shared memory, eliminating register pressure entirely. 4× wider than our 32-thread MMA.

📦

TMA

Tensor Memory Accelerator — hardware DMA that copies tiles from global to shared memory with zero thread involvement. Address generation, bounds checking, and swizzling all in silicon.

🧵

Warp Specialization

Dedicated TMA warps load data while math warps compute — true pipeline parallelism. ClusterTransactionBarrier synchronizes across warp groups with minimal overhead.

🧠

TMEM (sm_100)

Tensor Memory — a hardware-managed register file separate from the standard RF. B200's UMMA reads operands directly from TMEM, eliminating shared memory for the B matrix entirely.

09 — PTX MMA Register Layout

The layout that made it possible

mma.sync.aligned.m16n8k16 produces a 16×8 output tile. Each of the 32 threads holds exactly 4 float values at deterministic positions. This known layout is what enables in-register softmax — we can compute row-wise operations without ever touching shared memory.

Thread → Output Element Mapping
Each thread t holds: d0=C[row0,col0] d1=C[row0,col1] d2=C[row1,col0] d3=C[row1,col1]
row0 = (t/4)%8    row1 = row0+8    col0 = (t%4)*2    col1 = col0+1
■ Rows 0-7 (d0, d1) ■ Rows 8-15 (d2, d3)

Four threads share each row (varying in lane_id%4). A shuffle reduction with __shfl_xor_sync across delta 1 and 2 gives the partial max/sum for 32 columns. One tiny smem exchange between warp halves completes the full 64-column reduction.
